# 裸 API vs OpenCode 对比体会

## 实验过程

- **裸 API 调用**：直接调用 DeepSeek Chat API，传入纯文本 prompt `"请分析当前项目的目录结构和代码质量，给出改进建议。"`。模型返回了通用性的答案——它假设了一个典型的 Python Web 项目并给出泛化建议，完全没有感知到当前仓库的实际文件（`.gitignore`、`raw_api_test.py`、`README.md` 等）。回答结构工整但毫无针对性。

- **OpenCode 编排**：（待实验）OpenCode 会先调用 `read` / `glob` 工具浏览实际目录结构，读取关键文件内容，将真实上下文注入 LLM，再由 LLM 给出针对性的分析。

## 关键差异

| 维度 | 裸 API | OpenCode |
|------|--------|----------|
| **是否能读文件** | ❌ 不能。LLM 只看到 prompt 文本 | ✅ 能。通过 `read`、`glob`、`grep` 工具读取文件 |
| **是否有上下文感知** | ❌ 无。看不到目录结构、代码内容 | ✅ 有。感知真实目录树、文件内容、Git 历史 |
| **建议是否具体** | ❌ 泛化。套用模板化建议 | ✅ 具体。针对实际代码给出建议 |
| **能否操作项目** | ❌ 只生成文本 | ✅ 可以直接编辑文件、创建文件 |
| **多步推理** | ❌ 单次问答 | ✅ 可以多步工具调用、自我纠错 |

## 我对「无状态推理」和「有状态编排」的理解

无状态推理就像在面试中只看简历回答问题——LLM 只能靠训练数据中的"模板知识"来回答，看不到真实环境。有状态编排像是一个工程师带着 IDE 进场——先读文件、看目录、查 Git 历史，再动手改代码。OpenCode 通过工具系统（Tool System）把 LLM 从一个"文本生成器"变成了"能操作真实项目的 Agent"。这种区别本质上就是**信息是否闭环**：无状态推理的信息链路是断的（prompt → 回答），有状态编排的信息链路是闭环的（感知 → 推理 → 行动 → 验证）。
